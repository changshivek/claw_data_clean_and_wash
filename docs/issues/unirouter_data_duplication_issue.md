# Issue Draft: Unirouter 采集日志中存在会话快照重复，导致数据集前缀重复率偏高

## 1. 问题提出与说明

在对 unirouter 采集数据做抽样检查时，发现数据集中存在一类明显的“近重复”问题：

- 不是同一条完整请求被简单重复写入多次；
- 而是同一个会话在不同时间点的多个请求快照被同时保留；
- 这些快照共享相同的前序对话内容，只在后续追加轮次上有所不同；
- 如果直接把每条请求日志都当成独立样本使用，会导致训练/评测数据中出现大量高重叠前缀。

这类问题会带来几个直接影响：

- 放大某些会话主题的样本权重；
- 让数据集中的“有效独立样本数”低于表面样本数；
- 在下游训练或评测中引入偏置，因为模型会反复看到几乎相同的前序上下文；
- 如果去重只按 raw_json 或整条请求字符串做精确比较，这类问题不会被识别出来。

从这次排查结果看，问题根源更接近“采集粒度”而不是“导入重复”。也就是说，采集层把同一会话的递进式请求都保留成了独立样本，但这些样本之间存在很强的前缀重叠。

## 2. 数据举例

以下 3 条样本在抽查时被怀疑是重复数据：

- 原始文件：`request-logs-cmn9pe6bg00ksmoa266ms55rp-full-2026-03-28T03-32-29-012Z/items.jsonl`
- 样本 A：`log.id = cmn9raey900o0moa29lizs2lm`，`requestId = 212039d3-996b-4f35-a30f-a069c4f180a6`，位于 `items.jsonl` 第 `6` 行
- 样本 B：`log.id = cmn9rfuzv00o8moa27rgt1dz3`，`requestId = 649bb162-27b9-4e18-8d60-fe31989ebaad`，位于 `items.jsonl` 第 `2` 行
- 样本 C：`log.id = cmn9rc1z100o6moa2hafp7dzc`，`requestId = 9dc46df3-f99b-43b5-813b-d589378af8e4`，位于 `items.jsonl` 第 `3` 行

进一步核查后发现，它们并不是同一条记录的重复导入，而是同一会话中的不同请求快照：

- `userId` 相同：`cmmx1g9rh001dlj9yr7u8h1xg`
- `clientTokenId` 相同：`cmmx34lxl003vlj9yrrjgfjh6`
- `routePath` 相同：`/v1/internal/v1/messages`
- `modelRequested` 相同：`minimax-m2.5-fp8`

但它们的请求时间和请求内容推进阶段不同：

- 样本 A（`cmn9raey900o0moa29lizs2lm`）请求开始时间：`2026-03-28T03:13:48.880Z`，消息数 `17`
- 样本 C（`cmn9rc1z100o6moa2hafp7dzc`）请求开始时间：`2026-03-28T03:14:51.424Z`，消息数 `23`
- 样本 B（`cmn9rfuzv00o8moa27rgt1dz3`）请求开始时间：`2026-03-28T03:17:42.140Z`，消息数 `25`

针对样本 B（`log.id = cmn9rfuzv00o8moa27rgt1dz3`，`requestId = 649bb162-27b9-4e18-8d60-fe31989ebaad`），提取出真实用户文本后，可得到 3 轮核心用户输入：

1. `kilo code如何设置 将提供商设置成自己公司的url和自己的api`
2. `是vscode的一个插件，叫做kilo code`
3. 一段用于配置 Kilo Code 的环境变量 JSON（包含 `ANTHROPIC_BASE_URL`、`ANTHROPIC_AUTH_TOKEN`、`ANTHROPIC_MODEL` 等字段）

这说明该样本并不是一条孤立请求，而是同一个会话在逐步追问过程中的后期快照。

## 3. 统计标准说明

为了避免 system prompt、工具调用结果、assistant 思维文本等噪声干扰，这次统计没有直接比较整条 `messages` 或整条 `raw_json`，而是使用了两个更贴近真实内容的标准：

### 标准 A：真实用户内容抽取

从请求 payload 中仅保留：

- `role = user` 的消息；
- 其中 `type = text` 的文本块；
- 去掉 `Sender (untrusted metadata)` 之类的外包装；
- 去掉形如 `[Sat 2026-03-28 11:17 GMT+8]` 的时间戳前缀；
- 去掉会话启动提示，例如 `A new session was started via /new or /reset.`。

也就是说，统计对象是“真实用户表达的内容”，而不是请求日志的完整壳层。

### 标准 B：两类重复判断

#### B1. 最终问题近重复

对目标样本的“最后一轮真实用户输入”做 token 集合比较，使用 Jaccard 相似度统计原始数据中有多少样本与其高度相似。

这个标准用来回答：

“原始数据里是否存在大量与该样本最终问题几乎相同的记录？”

#### B2. 会话前缀重复

对目标样本前两轮真实用户输入做精确匹配，统计原始数据中有多少记录拥有完全相同的前两轮用户内容，但后续轮次不同。

这个标准用来回答：

“原始数据里是否存在大量同一会话在不同阶段的快照样本？”

这个标准比直接做整条消息字符串相似度更稳健，也更符合这次问题的本质。

## 4. 统计结果说明

### 4.1 精确重复检查

先按整条内容做精确检查，确认不是简单导入重复：

- 目标样本 B（`log.id = cmn9rfuzv00o8moa27rgt1dz3`，`requestId = 649bb162-27b9-4e18-8d60-fe31989ebaad`）在当前 DuckDB 中仅出现 `1` 次；
- 该样本对应内容在原始 source 中按 canonical JSON 也仅出现 `1` 次；
- 它在原始文件中的定位为：`request-logs-cmn9pe6bg00ksmoa266ms55rp-full-2026-03-28T03-32-29-012Z/items.jsonl` 第 `2` 行。

结论：

- 没有证据表明存在“同一条 raw_json 被重复导入多次”；
- 问题不在导入层的 exact duplicate；
- 问题在于“同一会话不同阶段的快照都被采集下来”。

### 4.2 最终问题近重复

对目标样本 B（`cmn9rfuzv00o8moa27rgt1dz3` / `649bb162-27b9-4e18-8d60-fe31989ebaad`）的真实用户内容做统计后：

- 原始数据中有真实用户轮次的记录总数：`22492`

按“最后一轮真实用户内容”的 token/Jaccard 相似度统计：

- `>= 0.95`：`1`
- `>= 0.90`：`1`
- `>= 0.80`：`1`
- `>= 0.70`：`1`

结论：

- 就“最终问题文本”而言，这条样本在原始数据中并没有大规模近重复；
- 如果只看用户最后一轮具体诉求，这个 case 不算严重。

### 4.3 会话前缀重复

对目标样本前两轮真实用户输入做精确匹配，结果如下：

- `SAME_FIRST_TWO_COUNT = 38`
- `SAME_ALL_THREE_COUNT = 1`
- `SAME_FIRST_TWO_BUT_DIFFERENT_LATER_COUNT = 37`

这 3 个数字的含义是：

- 原始数据中共有 `38` 条记录拥有和目标样本 B 完全相同的前两轮用户输入；
- 其中只有 `1` 条和目标样本三轮都一致，也就是样本本身；
- 剩余 `37` 条都属于“前两轮相同、但后续发展不同”的同会话快照。

其中一部分可直接在原始文件中定位，例如：

- `request-logs-cmn9pe6bg00ksmoa266ms55rp-full-2026-03-28T03-32-29-012Z/items.jsonl` 第 `1` 行：`log.id = cmn9rwik900oamoa2qgbappdh`，前两轮相同，但后续多出新的第三轮用户输入；
- 同一文件第 `3` 行：`log.id = cmn9rc1z100o6moa2hafp7dzc`，只保留到前两轮；
- 同一文件第 `6` 行：`log.id = cmn9raey900o0moa29lizs2lm`，也属于同一会话更早阶段的快照；
- 另外在 `request-logs-cmn9pe6bg00ksmoa266ms55rp-incremental-2026-03-29T04-00-21-616Z/items.jsonl` 中，也能看到同一前两轮被后续更长会话链继续引用的样本。

结论：

- 这类问题在原始数据中是明显存在的；
- 重复的不是“完整任务”，而是“同一会话的公共前缀”；
- 如果下游把这些快照都当作独立样本，会显著放大某些对话前缀的权重。

## 5. 修复建议

建议把修复重点放在**采集层**，而不是只依赖离线去重。

### 建议 1：引入会话级主键，避免把递进快照直接当作独立样本

采集侧应优先使用稳定的会话标识做聚合，例如：

- `clientTokenId`
- `sessionId`（如果有）
- `userId`
- 以及时间顺序字段，例如 `requestStartedAt`

其中最直接的方向是：

- 如果同一个 `clientTokenId` / `sessionId` 下，新的请求只是对已有会话追加上下文，优先视为“同一会话的增量”，而不是一个全新的独立样本。

### 建议 2：采集时按会话做累加存储，而不是每次完整展开存一份

当前问题的本质是每次请求都把历史对话完整展开后再次落盘。

更合理的采集方式可以是：

- 存储会话级对象；
- 每次只记录新增轮次或增量 diff；
- 导出训练/评测样本时，再根据需要重建指定阶段的完整上下文。

这样能从源头避免“大量共享前缀的快照样本”。

### 建议 3：在采集层增加前缀快照去重策略

如果当前系统必须保留逐次请求日志，也建议增加一层轻量去重/折叠逻辑：

- 当新请求和上一个请求属于同一 `clientTokenId` / `sessionId`；
- 且前 `N` 轮用户内容完全相同；
- 仅在最后若干轮发生追加；

则可将其标记为：

- `conversation_snapshot = true`
- `snapshot_parent = <previous_request_id>`
- `snapshot_depth = <k>`

这样即使不直接删除，也能让下游在构建数据集时选择：

- 只保留最终快照；
- 只保留关键阶段；
- 或对同一快照链降权采样。

### 建议 4：离线导出时增加会话链折叠规则

如果采集层暂时无法改动，至少应在导出训练/评测集前做一轮会话链折叠：

- 先按 `clientTokenId` / `userId` / 时间排序构造会话链；
- 再检测“前两轮或前 N 轮相同，但后续递进”的样本链；
- 每条链只保留一个代表样本，或者只保留最后一个快照。

### 建议 5：在元数据中显式区分“完整样本”和“会话快照”

建议未来新增字段，例如：

- `is_incremental_snapshot`
- `conversation_id`
- `parent_request_id`
- `snapshot_index`
- `cumulative_turn_count`

这样下游使用者不需要重新反推，就能直接识别哪些记录只是同一会话的中间快照。

---

## 6. 建议的 issue 结论

建议将该问题定义为：

> unirouter 当前采集逻辑会把同一会话在不同阶段的请求快照重复保留下来，导致数据集中出现大量共享前缀的近重复样本。问题主要不是 exact duplicate，而是 conversation snapshot duplication。建议在 client/session 维度做会话聚合、增量存储或快照折叠，避免下游数据集被高重叠前缀放大。

这个结论和当前排查证据是一致的，也足够具体，便于 unirouter 项目按采集链路定位与修复。
